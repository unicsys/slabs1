üí° Understanding the Flow

The diagram above shows exactly what happens when a user queries "Show all defects for Engine E1234":

  1. User asks a question using business terminology (not database jargon)
  2. Semantic layer consults the ontology to understand what "AircraftEngineIdentifier" means
  3. Mapping registry reveals that this concept exists in 3 different databases with different column names
  4. Query translator generates 3 SQL queries automatically - one for each database
  5. Queries execute in parallel against the physical databases
  6. Results are merged using ontology relationships (they all reference the same engine)
  7. User receives a unified view without knowing data came from multiple sources

mappings = registry.get_mappings("AircraftEngineIdentifier")

# - AssetTracker_DB.SERVICE_RECORDS.Asset_Serial
3 Generate SQL Queries 
sql1 = "SELECT ENGINE_SN, DEFECT_DESC FROM MaintDB_East.ENGINE_DEFECTS WHERE ENGINE_SN = 'E1234'"
sql2 = "SELECT Engine_ID FROM WorkOrderDB.MAINTENANCE_LOG WHERE Engine_ID = 'E1234'"  
sql3 = "SELECT Asset_Serial FROM AssetTracker_DB.SERVICE_RECORDS WHERE Asset_Serial = 'E1234'"
4 Execute & Merge Results 
results = []
for sql in [sql1, sql2, sql3]:
    results.extend(execute_sql(sql))

unified = merge_by_identifier(results)  # Use ontology to join correctly
5 Return Unified Results
